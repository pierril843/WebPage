Elevator Project

2017 - 3rd Year Project

The elevator project consisted of controlling a miniature elevator over a CAN network, interface the CAN network to a local network (apache on Supervisor node) and create a front end website that allows diagnostic controls. In total the network had 6 nodes; all nodes except the supervisor is a HCS12-based board:

- 1-3 Floor Nodes - Each floor has a button for requesting the elevator via CAN, and LEDs would keep track of which floor the elevator is on by monitoring the messages sent by the EC.

- Car Node - Used as the inside portion of the elevator; handles the input of floor destination along with opening, closing the doors (simple state no control to actual doors) and displaying current floor with the same LED system as floor nodes.

- Elevator Node - this controller interfaces with the motors and position sensor. Listens for commands from the Supervisor and reports current car position to all nodes.

- Supervisor Node - Linux host communicating with PCAN; acts as the elevator state machine. It receives commands from the floor and car nodes along with the web interface. When moving to a new floor the supervisor would be the node responsible for telling the Elevator node to move. The supervisor alongside apache ran a MySQL instance to keep track of requests and the current state in the state machine. On the website, php was used to modify and make requests to the database. Javascript was used to give a live update of the car along with the recent request logs.

    2016 - 2nd Year Project

    Overview: This programmable HCS12 based microcontroller project was the pride of the Embedded system engineering program and would go one to become the backbone of future projects. Designed by each student in 2nd year, this microcontroller uses an HCS12 processor as its core, and housed the circuitly and pinouts for many input/output interfaces.

    Capabilities: A power source of 12V DC is stepped down to to serve as the supply for a safe 5 volts. The controller supplies pinouts for a servo and stepper motor making it very useful on later projects that require low resistance motion. it is capable of driving two dc motors directly and to add to the accuracy there are two sets of pinouts that are reserved for a pair of encoders, allowing for software PID to control speeds/voltage output.

    Communication: This controller is able to communicate with the outside work via a couple different means. Outside of the motor drivers it can power and edit the text of an added LCD display (16x2) to display system updates and changes. This would go on to be very useful for light debugging and live error or warning to be displayed. So that covers output but are there any input methods? Yes we also allow you to attach a mic to make use of onboard ADC (modification can be made to allow any number of analog signals to be encoded). Though a controller with no way to send live commands or instructions would be neat but not useful for all situations so there is also the ability to communicate using RS-232 cable. Communication over the RS-232 line would be a custom protocol depending on the tasks and size requirements for data. The addition of error detection was added to many projects to help with line signal degradation.

    Debugging: The ability to step through code on the live target is very useful so we of course would tap into the uncluded functionality of the MCUâ€™s BDM port. In attadition the software used (codewarrior) allowed use to simulation running tasks on they NXP MCU (HCS12) before flashing the target.
